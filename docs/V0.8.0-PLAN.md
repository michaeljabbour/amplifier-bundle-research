## INCIDENT LOG — v0.8.0 + v0.8.1 superseded by v0.8.2

**Date:** 2026-04-30
**Severity:** Recipe schema violations (5 of 5 new recipes fail validation) + 1 active agent name collision

**What was broken in v0.8.0/v0.8.1:**
- All 5 new recipes shipped in v0.8.0/v0.8.1 fail `recipes operation=validate` due to fabricated step-type primitives (`type: while`, `type: llm_judge`, `type: shell`), invented kwargs (`inputs:`, `outputs:` plural, `body:`, `initial_state:`), and silently-dropped flat `approval_required:` keys (which disabled the human-in-loop approval gates without erroring).
- `agents/paper-architect.md` `meta: name` collided with `amplifier-bundle-scientificpaper`'s identical agent name. When both bundles were installed simultaneously, the resolver silently picked one.
- Top-level `cache_only:` and `judge_cross_vendor_required:` keys on the 9 baseline recipes were silently ignored by the recipe engine (these belong inside each recipe's `context:` block to be reachable as substitutable variables).

**Detection:** `recipes:recipe-author` and `amplifier:amplifier-expert` agents ran a post-ship audit at user request (the user explicitly asked "did you check the orchestrator doesn't conflict with foundation/standard bundles?"). All issues were surfaced before any external user pulled the broken tags.

**Fix:** v0.8.2 (this release).
- `paper-architect` renamed to `research-paper-architect` (collision broken).
- `orchestrated-loop.yaml` rewritten in the canonical `attractor.yaml` Stage 3 pattern (one `type: recipe` step with `while_condition + max_while_iterations + update_context`); iteration body extracted into new `orchestrated-loop-iteration.yaml` sub-recipe.
- All `inputs:` → `context:`, all plural `outputs:` → singular `output:`, all flat `approval_required: + approval_prompt:` → nested `approval: { required, prompt, timeout, default }`.
- `cache-only-verify.yaml` `type: shell` → `type: bash`; Jinja-style filters removed.
- `cache_only:` and `judge_cross_vendor_required:` moved INTO each recipe's `context:` block.
- New `scripts/v0.8.2-healthcheck.sh` adds a recipe-schema-validation gate to prevent recurrence.

**Tag handling (Option B, per user direction 2026-04-30):**
- Tags `v0.8.0` (commit `a479c3f`) and `v0.8.1` (commit `01a59cf`) are LEFT IN PLACE on GitHub as broken-historical record.
- Tag `v0.8.2` is the canonical release going forward.
- GitHub release notes for v0.8.0 and v0.8.1 are flagged "SUPERSEDED — see v0.8.2 for canonical recipe schemas. Do not use these tags for production."
- This matches the v4d → v4e → PSE-bundle posture: name the broken artifact explicitly, ship the fix forward, do not force-move tags.

**Out of v0.8.2 scope (documented as known issues):**
- 8 of the 9 baseline recipes (`empirical-paper.yaml`, `benchmark-paper.yaml`, `replication-study.yaml`, `literature-review.yaml`, `patent-brief.yaml`, `policy-brief.yaml`, `white-paper.yaml`, `grant-proposal.yaml`) ALSO fail `recipes operation=validate` because they have no `steps:` or `stages:` — they're persona/spec documents structured as recipes that have lived in `recipes/` since v0.2.0 (April 2026). This pre-dates v0.8.0 entirely and is tracked separately as a known issue. v0.8.2 only fixes what v0.8.0/v0.8.1 broke.
- The hooks-mode `search_paths` deep-merge interaction with `amplifier-bundle-superpowers` is documented in `behaviors/research-modes.md` and is a pre-existing architectural hazard that requires a wrapper bundle to resolve.

**Lesson learned (process change for future releases):**
Authoring a new recipe in this bundle now requires `amplifier tool invoke recipes operation=validate recipe_path=<path>` returning success BEFORE the file is committed. The new `scripts/v0.8.2-healthcheck.sh` checks for the structural patterns that the validator catches plus the silent-failure patterns it doesn't. Both gates are now part of the bundle's release-readiness contract.

## v0.8.3 — closure of remaining pre-existing bugs + new test gate suite

**Date:** 2026-04-30
**Triggered by:** user request after v0.8.2 ship — "fix the other 2 bugs you found? any other tests we want to add?"

**What v0.8.3 closed:**

### Bug 1: 8 baseline recipes failing `recipes operation=validate` (pre-existing since v0.2.0)
The 8 recipes (`empirical-paper`, `benchmark-paper`, `replication-study`, `literature-review`, `patent-brief`, `policy-brief`, `white-paper`, `grant-proposal`) had no `steps:` or `stages:` blocks — they were persona/spec documents with `modes:`, `defaults:`, `behaviors:`, `persona_signals:`, `publish_gates:` blocks. Validator rejected all 8 with `"Recipe must have at least one step or stage"`.

**Fix:** all 8 rewritten by `recipes:recipe-author` to canonical `stages:` format. ALL VALID. Total expansion: 1,110 → 3,861 lines (+2,751). Original documentation blocks PRESERVED as design contracts under a "DESIGN-CONTRACT DOCUMENTATION" banner — they survive as engine-ignored top-level keys.

**Schema discoveries that surfaced during the rewrite (carry-forward for future work):**
- The schema rejects `class:` inside `provider_preferences:` entries. Cross-vendor enforcement is via explicit `provider:`/`model:` fallback chains across vendor families, NOT `class: reasoning`.
- The schema rejects `model_role:` inside `provider_preferences:` entries. `model_role` is a step-level field, not a provider-preference field.
- A step cannot declare both `model_role:` and `provider_preferences:` simultaneously — validator emits `"cannot use both 'model_role' and 'provider_preferences' - choose one approach"`.
- Top-level keys not in the schema (`modes:`, `defaults:`, `behaviors:`, `persona_signals:`, `publish_gates:`, `special_behavior:`, `honest_pivot_mode:`) are silently accepted as documentation.

`behaviors/cross-vendor-judge.md` was updated to reflect this schema reality (the earlier draft incorrectly described `class: reasoning` as the canonical form).

### Bug 2: engine boolean-string coercion runtime risk
The validator accepts `condition: "{{enable_no_op_arm}} == 'true'"` but the runtime behavior depends on whether the engine stringifies YAML bool `true` as `"True"` or `"true"`. v0.8.2 applied a defensive stringification fix (context vars as `"true"` strings), but that fix had no test.

**Fix:** new internal smoke-test recipe `recipes/_smoke-boolean-condition.yaml` (the underscore prefix marks it as bundle-internal and excludes it from the validate-all-recipes scan). The recipe has 4 conditional steps testing each combination of YAML bool / explicit string and `==`/`!=`. Run via `recipes operation=validate` to confirm parse-clean; full execution (with real agents) would confirm the runtime semantics match the validator's acceptance.

### New test gate suite (preventing this class of bug from recurring)

Five new test gates added in v0.8.3:

| Gate | What it catches |
|---|---|
| `scripts/validate-all-recipes.sh` | Every recipe must pass `recipes operation=validate`. Slow (one CLI invocation per recipe); exclude with `--skip-recipes-validate` for fast iteration. The gate that v0.8.0/v0.8.1 healthcheck didn't have. |
| `scripts/check-agent-collisions.sh` | Scans every `~/dev/amplifier-bundle-*/agents/*.md` for `meta:name` collisions. Fails if research is involved. |
| `scripts/check-recipe-agent-refs.sh` | Every `agent: research:NAME` reference in `recipes/` must resolve to an actual agent file. |
| `scripts/check-behavior-includes.sh` | The composite behavior `behaviors/research.md` include list must match the actual files in `behaviors/`. |
| `tests/test_calibration.py` (extended) | New `test_cli_calibration_end_to_end` exercises the `amplifier-research-power calibration` CLI subcommand end-to-end via subprocess. 16 unit tests + 1 CLI = 17 total. |

`scripts/v0.8.3-healthcheck.sh` wires all of them together.

### Real bugs the new gates caught (immediate value)

- **`scripts/check-recipe-agent-refs.sh`** caught 4 stale `research:paper-architect` references in the recipe-author rewrites (recipe-author used the OLD agent name). FIXED via `sed`.
- **`scripts/check-agent-collisions.sh`** caught 2 pre-existing collisions involving research:
  - `citation-manager` collided with `amplifier-bundle-scientificpaper` (gene-transferred surplus from v0.8.0)
  - `technical-writer` collided with `amplifier-bundle-stories`
  - **Fix:** renamed to `research-citation-manager` and `research-technical-writer`. File rename + `meta:name` update + `bundle.md` include update + every recipe reference updated. Pattern matches the `paper-architect` → `research-paper-architect` rename from v0.8.2.
